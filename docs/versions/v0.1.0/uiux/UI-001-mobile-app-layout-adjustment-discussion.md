# UI-001 手机端 App 化页面调整讨论稿

> 状态：待确认
> 阶段：S2 方案设计 / UIUX 子流程
> 日期：2026-06-05
> 适用范围：TalkMate 手机端优先体验调整

## 1. 背景与问题

当前 TalkMate 已声明移动端优先，但页面实现仍带有明显 Web 页面特征：

- 页面使用大面积居中卡片、宽屏容器、桌面式分区，缺少手机 App 的连续任务流。
- 登录、首页、对话页、总结页更像响应式网页，而不是原生移动应用界面。
- 核心操作没有固定在拇指容易触达的位置。
- 页面信息层级偏“展示型”，不够“任务驱动型”。
- 训练、反馈、复练等新表达已经出现，但缺少统一的 App 信息架构。

## 2. 调整目标

将 TalkMate 从“移动端可用网页”调整为“手机端优先的 AI 口语训练 App 页面”。

核心目标：

- 用户打开后能像使用 App 一样完成登录、选择任务、开始对话、生成反馈、复练。
- 375px 视口作为主设计目标，768px 和 1440px 仅做适配扩展。
- 页面采用底部操作、单列任务流、清晰状态反馈、稳定导航。
- 保留 Web 技术栈，不引入 Native App 或复杂跨端框架。

## 3. 推荐方向

推荐采用“移动 App Shell + 任务训练流”方案。

### 3.1 全局 App Shell

- 页面外层统一为手机 App 壳层：`min-h-dvh`、单列、最大内容宽度建议 `430px-480px`。
- 桌面端只居中展示手机宽度内容，不再拉成大屏网页。
- 顶部使用轻量状态栏式导航：品牌、当前任务、用户入口。
- 底部使用固定主操作区：开始任务、发送消息、结束训练、查看反馈。
- 避免大面积 hero、宽屏网格、桌面 dashboard 式布局。

### 3.2 登录 / 注册页

调整为 App 启动页，而不是网页登录卡片：

- 顶部：品牌与一句明确定位，例如“AI 英语口语训练”。
- 中部：表单单列，输入框高度不低于 48px。
- 底部：主按钮固定或靠近底部，便于单手操作。
- 注册页不做过多营销文案，减少滚动负担。
- 错误提示贴近字段，整体错误可放在按钮上方。

关键状态：

- 默认：表单可输入。
- Loading：按钮禁用，显示处理中。
- Error：字段级错误 + 页面级错误。
- Disabled：未满足输入条件时主按钮不可用。

### 3.3 首页

调整为 App 首页 / 训练任务页：

- 顶部欢迎区只保留用户状态与今日建议，不做桌面 hero。
- 主体第一屏优先展示“今天推荐训练”与“5 个场景任务”。
- 场景卡片改为移动任务卡：任务名、AI 角色、训练目标、预计时长、开始按钮。
- 历史记录放在任务卡之后，用紧凑列表展示。
- 桌面端不改成 3 列大网格，仍保持手机 App 内容宽度，可在外围留白。

关键状态：

- 场景 Loading / Empty / Error。
- 历史记录 Loading / Empty / Error。
- 登录过期跳转登录。

### 3.4 对话页

调整为 App 核心训练页面：

- 顶部固定：返回、场景/任务标题、进度。
- 中部：消息列表全屏滚动，用户与 AI 气泡适配单手阅读。
- 底部固定：语音按钮、文本输入、发送按钮。
- 结束训练按钮不应挤在顶部宽屏布局中，可放在顶部次操作或底部更多操作中。
- 录音态必须强反馈：录音中、识别中、识别失败、权限被拒绝。
- 历史回放模式必须和训练模式明显区分，禁用输入区。

关键状态：

- 新对话空态。
- 消息发送中。
- AI 回复中。
- STT 录音中 / 失败 / 权限拒绝。
- TTS 播放中 / 不支持。
- 发送失败可重试。
- 历史回放只读态。

### 3.5 总结页

调整为 App 训练报告，而不是网页报告页：

- 第一屏展示评分、任务完成度、最重要的 1-3 个问题。
- 纠错内容按卡片折叠或分段，避免长列表压迫。
- 明确“下一步行动”：复练本句、回到对话、换任务。
- 中式英语画像可以保留，但必须作为反馈模块，不要喧宾夺主。
- 底部固定主 CTA：复练 / 选择下一任务。

关键状态：

- 生成中。
- 生成失败可重试。
- 无总结。
- 无纠错但有总结。
- 历史总结查看。

## 4. 视觉与组件原则

- 宽度：移动主容器以 375px 为核心，最大宽度建议 430px 或 480px。
- 圆角：App 卡片建议 16px-20px，按钮 12px-999px 视用途统一，不要每页随意变化。
- 主色：Indigo 仅用于主操作、重点标签和焦点态，减少大面积渐变背景。
- 导航：统一顶部导航和底部主操作，不按页面各自发挥。
- 文案：使用“任务、训练、反馈、复练”等稳定词汇，避免“网页营销式”大标题。
- 触控：所有关键按钮高度不低于 44px，输入区不低于 48px。
- 安全区：底部固定操作必须考虑移动浏览器地址栏和安全区。

## 5. 页面调整优先级

| 优先级 | 页面 | 调整重点 |
|---|---|---|
| P0 | 登录页 | App 启动页、底部主按钮、字段错误 |
| P0 | 注册页 | 单列短流程、输入校验、底部主按钮 |
| P0 | 首页 | 任务流首页、推荐训练、移动任务卡、历史列表 |
| P0 | 对话页 | 顶部进度、全屏消息、底部固定输入、语音状态 |
| P0 | 总结页 | 评分首屏、问题摘要、复练 CTA、长内容折叠 |
| P1 | 历史回放 | 只读态、总结入口、与训练态区分 |

## 6. 建议实施阶段

### 阶段 A：S2 方案确认

- 确认是否采用“手机 App Shell + 任务训练流”。
- 更新 UIUX 设计说明。
- 明确是否保留当前训练/成长反馈方向。

### 阶段 B：S3 任务拆解

- 拆成 App Shell、Auth、Home、Conversation、Summary、History、状态与测试七组任务。
- 每组任务明确完成标准和验证方式。

### 阶段 C：S4 开发实施

- 先建立全局 App Shell 和布局约束。
- 再逐页调整，不同时做业务扩展。
- 禁止把视觉调整、训练功能扩展、接口变更混在一个提交。

### 阶段 D：S5 联调验证

- 执行移动端主流程、状态覆盖、多视口、构建和回归测试。
- 收集截图、日志和命令输出证据。

## 7. 验收测试矩阵

### 7.1 自动化基础验证

| 类型 | 命令/方式 | 目标 |
|---|---|---|
| 前端构建 | `npm --prefix frontend run build` | TypeScript 与 Vite 构建通过 |
| 后端模块测试 | `pytest` 既有 auth/scenario/conversation/summary/ai_service 用例 | 确认 UI 调整未破坏接口契约 |
| E2E 主流程 | Playwright | 注册登录 → 首页 → 对话 → 总结 → 历史回放 |

### 7.2 移动端视觉验收

必须覆盖视口：

- 375x812：主验收视口。
- 390x844：常见 iPhone 视口。
- 430x932：大屏手机。
- 768x1024：平板兼容。
- 1440x900：桌面容器居中展示，不出现网页化拉伸。

每个视口检查：

- 无水平滚动。
- 关键按钮高度不低于 44px。
- 底部操作不被浏览器安全区遮挡。
- 文本不溢出、不互相遮挡。
- 主操作在拇指可触达区域。
- 页面不像桌面网页被压缩，而是手机 App 流式布局。

### 7.3 页面状态验收

| 页面 | 必测状态 |
|---|---|
| 登录 | 默认、输入错误、登录中、登录失败、登录成功跳转 |
| 注册 | 默认、密码不一致、验证码错误、注册中、注册成功 |
| 首页 | 场景加载中、场景为空、场景失败、历史为空、历史失败、正常列表 |
| 对话 | 新对话空态、发送中、AI 回复中、发送失败、录音中、录音失败、TTS 播放中、历史只读 |
| 总结 | 生成中、生成失败、无总结、无纠错、正常总结、复练入口 |

### 7.4 核心业务验收

- 用户能完成注册、登录、刷新保持登录、登出。
- 用户能在手机视口选择 5 个场景任务。
- 用户能通过文字或语音完成至少 10 轮训练。
- AI 回复能展示并可播放。
- 结束训练后能生成总结。
- 总结页能展示评分、纠错、词汇分析、改进建议。
- 历史记录能回放完整对话并进入总结。

### 7.5 证据要求

- Playwright 截图：每个核心页面至少保留 375px、430px、1440px 截图。
- 测试报告：记录用例、步骤、实际结果、结论、证据路径。
- 命令输出：前端 build、后端 pytest、E2E 执行结果。
- 缺陷记录：发现问题必须按 Issue 格式记录并复测。

## 8. 门禁结论

当前结论：需调整。

原因：

- 用户已明确指出页面不像手机 App，现有实现需要回到 S2 UIUX 子流程补齐设计。
- 当前可先确认移动 App Shell 方向，再进入 S3 任务拆解。
- 未经确认前，不应继续直接修改页面实现。
